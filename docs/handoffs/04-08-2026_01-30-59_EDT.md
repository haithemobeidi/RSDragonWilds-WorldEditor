### 2. 🎯 PROP[2] counter — the long-standing mystery solved

The `GlobalBuildingManager` NOBJ has a PROP sub-chunk with 4 properties:

- prop[0]: 12 bytes (int64 config = 14)
- prop[1]: 12 bytes (int64 config = 15)
- **prop[2]: 4 bytes (uint32 — per-world counter)**
- prop[3]: 1 byte (bool = true)

Observed values:
- TransplantTest with 1 placed piece: counter = 1
- DiffTest with ~45 placed pieces: counter = 50
- After successful v3 transplant setting counter = 50: cabin rendered

Exact semantics still unclear — could be "next persistent_id to assign", "total pieces ever placed", or "max pid seen". Whatever it is, it MUST be high enough to cover the IDs of all transplanted pieces or the game silently rejects them on load.

### 3. 🎯 Category 1 vs Category 2 structure split — verified with hex evidence

**Category 1 (passive mesh pieces — walls, floors, roofs, doors):**
- Stored ONLY in the Pces chunk inside `GlobalBuildingManager.CUST`
- Game resolves the base64 class GUID from the `.pak` global class registry at runtime
- No SPWN record needed
- Pces-only transplant works

**Category 2 (interactive actors — chests, furnaces, crafting stations):**
- Stored in BOTH the Pces chunk (placement) AND a SPWN chunk inside `L_World.SATS` (actor state)
- The Pces record says "a chest should be here", the SPWN record has the actor with inventory, owner, health, etc.
- Pces-only transplant fails silently — game reads the Pces placement, tries to look up the matching SPWN actor, doesn't find it, gives up

**Verified via hex byte search:** Personal Chest 1's position doubles `(8553.33, 185509.18, -3230.51)` appear at:
- Offset `0xb342` — inside the Pces chunk
- Offset `0xef3f` — inside a 589-byte SPWN chunk at `0xeeab-0xf100` (inside `L_World.SATS`)

Same 24 bytes, two locations. The chest is stored twice, in two different chunk types, and both must be transplanted together for the chest to actually appear in-game.

### 5. 🎯 Empirical format quirks discovered

- **LEVL body has an 8-byte version header after the FString name** (`0a 02 00 00 f9 03 00 00` = two uint32s: 522 and 1017). Undocumented in SPUD source. Walker must skip these.
- **NOBJ body has ClassID at the START** (before the FString Name), not the end as SPUD's header file suggests.
- **The 8-byte `0a 02 00 00 f9 03 00 00` sequence is a universal SPUD marker** — appears at the start of LEVL bodies, NOBJ bodies, and inside Pces records at offset 0x44.
- **Pces is NOT a SPUD primitive** — it's a Dragonwilds-custom chunk nested inside a SPUD CUST chunk inside a NOBJ.

### 6. 🎯 Spatial collision loss theory for "missing walls"

During the cabin transplant, the user's 2 dupe walls disappeared. Theory: the walls were at Z=-3254 while the cabin floor was at Z=-3187 (67 cm higher), and the walls were within the cabin's X/Y footprint. When the game instantiated the cabin floor tiles, they overlapped with the walls spatially, and the game resolved the collision by silently dropping the walls.

Not 100% confirmed — user noted "we're assuming it's collision but idk." Treated as a hypothesis in the guide with a warning recommendation.

### 7. ✅ World rename with chunk-length fixup

Built `rename_world.py` that renames a world's internal `WorldName` and `WorldSlotName` from one string to another of different length, updating all containing chunk lengths (SAVE, INFO, CINF, GLOB, GOBS, NOBJ) via a strict whitelist-based walker that avoids phantom chunks from byte-scanning random data.

First version had a bug — it recursed into non-container chunks and found false-positive phantom chunks, then corrupted 3 bytes in the file by decrementing fake "length" fields. Fixed by adding a strict `VALID_TAGS` whitelist that filters phantom chunks.
