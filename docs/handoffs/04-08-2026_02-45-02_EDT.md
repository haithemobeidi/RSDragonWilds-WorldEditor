## Accomplishments This Session

### 1. 🎯 Surgical transplant packaged into parser.py

Added to `WorldSave` class (~290 lines):
- `_find_gbm_nobj_layout()` — recursive chunk walker that returns all offsets needed for surgical modification (SAVE, LVLS, LEVL, LATS, NOBJ, CUST, Pces, PROP[2] counter offset). Handles the empirical quirks (LEVL body's 8-byte version header after the FString name, NOBJ ClassID at the start not end).
- `get_pces_body()` — returns the raw bytes of the GBM's Pces chunk
- `get_pces_counter()` — returns the GBM PROP[2] uint32 counter value
- `transplant_structures_from(source: WorldSave, auto_save: bool = True)` — the main feature. Reads source's Pces body, appends to target's Pces, updates 7 chunk lengths up the tree (Pces, CUST, NOBJ, LATS, LEVL, LVLS, SAVE), updates CUST TArray count, updates GBM PROP[2] counter to max(target, source). Returns a result dict with bytes added, before/after sizes, counter values.
- `_raw_save()` — writes raw_data directly to disk with auto-backup, used by binary-modifying operations that don't want the JSON-section-rewriting save() path.

### 2. 🎯 Dedicated world editor Flask app (world_editor.py)

A standalone, focused Flask app on port 5001 that does ONLY world editing. Doesn't share state with the main app on port 5000. ~370 lines.

**Routes:**
- `GET /` — render the single-page UI
- `GET /api/worlds` — list all worlds with summaries (filters out `EnhancedInputUserSettings.sav` which is an Unreal input settings file, not a world)
- `GET /api/world/<filename>` — full world details (header, piece counts, piece breakdown with friendly names)

**Friendly piece names:** added a `KNOWN_PIECE_NAMES` dict that maps base64 piece GUIDs to human-readable names ("Cabin Wall", "Personal Chest", "Square Floor Tile", etc.). Building piece classes aren't in our `guid_map.json` from the .pak extraction (which has items/quests), so this is a small empirical map built up during reverse engineering. Easy to extend.

## Technical Notes

1. **The world editor is intentionally separate from the main app.** Different port (5001 vs 5000), different template, different routes. They can both run at the same time. This keeps the world-editing UX focused without dragging in the bloated character/database tabs from the main app.

2. **Path resolution uses `_paths.py` helper.** The world editor's `_spud_cache_dir()` lazily imports from `scripts/structure_research/_paths.py` to keep the path detection logic DRY. Override via `RSDW_SAVES_DIR` and `RSDW_CACHE_DIR` env vars.

3. **Auto-backup before any destructive write.** The `transplant_structures_from()` method calls `_raw_save()` which writes a timestamped backup to `editor_backups/` before overwriting. The clone endpoint doesn't need a backup (it writes a NEW file, doesn't modify source).

4. **SpudCache awareness.** Both transplant and clone endpoints automatically clear `SpudCache/L_World.lvl` after their write. This prevents the game from loading stale cached data on the next launch.

5. **The clone endpoint's WorldName length delta handling** is tricky. If old name and new name have different lengths, every chunk containing the WorldName FString needs its length field updated. The walker uses a strict whitelist of valid SPUD tags (`VALID_TAGS`) and only recurses into known container tags (`RECURSE_TAGS`) to avoid the phantom-chunk bug we hit in the first version of `rename_world.py`.

## User Testing Results — IMPORTANT FINDINGS

User did a real in-game test of the surgical transplant via the world editor. Discoveries:

1. **World mode doesn't matter for transplants.** User transplanted from a Custom world INTO a Standard world successfully. The cabin pieces showed up. So our concern about world mode compatibility was unfounded.

2. **The Category 1 vs Category 2 split is more nuanced than we thought.** Refined model:

   - **Category 1A — Pure static meshes (full collision, no interaction):** walls, floors, roofs (likely — TBD on collision). These work completely after transplant.
   - **Category 1B — Visible but broken collision/interaction:** doors and water barrels show up visually but have NO collision (you walk through them) and can't be used. They need additional component state we don't transplant.
   - **Category 1C — Partially functional:** farming plots render and can be dug despite the transplant, suggesting their state requirements are minimal.
   - **Category 2 — Invisible / silently dropped:** chests, anvils, rune altars, presumably all "machines" / crafting stations / inventory-bearing actors. These need full SPWN+inventory record copying that we don't do. User confirmed: "my anvil and rune altar are missing as well — probably same reason the chests and other machines are missing".

3. **The "category" determination probably depends on what additional NOBJ/PROP/CORA/SPWN records the piece needs to bind.** Pure mesh pieces work because they're just instantiated from the Pces record. Pieces with separate collision components, interaction components, or inventory state need those components copied too — and we currently only copy the Pces placement metadata.

**Need to verify in next session by asking the user about wall/floor/roof collision specifically.** If walls block the player but doors don't, that confirms the "static mesh embeds its own collision; interactive pieces need separate collision components" theory.

**This is gold for the developer guide.** Once we have the collision data confirmed, we can write up the precise "what survives a Pces-only transplant" matrix.
