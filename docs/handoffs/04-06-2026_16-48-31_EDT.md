## 🎯 Major Discoveries This Session

### Discovery #1 — The Difficulty "Secondary Copy"

**Problem:** Phase 1 editor wrote 0.0 to the named TagName/NameProperty entries at offsets ~0x6dd, ~0x73d in Middle Eearth.sav. Disk file showed 0.0 ✅. Game launched, showed 0.5 ❌. Game saved, file reverted to 0.5/0.5/1.0.

**Root cause:** The named entries are NOT what the game reads. There's a **second, tagless float array** stored much earlier in the file, immediately after the `L_World\0` string. The game reads from THIS array. Our edit to the named entries was completely ignored.

**Exact structure (Middle Eearth):**
```
Offset 0x026d:  4c 5f 57 6f 72 6c 64 00     "L_World\0"
Offset 0x0275:  00                          1 byte padding
Offset 0x0276:  03 00 00 00                 count = 3 (uint32 LE)
Offset 0x027a:  01 00 00 00                 version = 1 (uint32 LE)
Offset 0x027e:  00 00 80 3f                 float[0] = 1.0  (NoBuildingStability)
Offset 0x0282:  00 00 00 3f                 float[1] = 0.5  (BuildingMaterialCostScale)
Offset 0x0286:  00 00 00 3f                 float[2] = 0.5  (CraftingCostScale)
Offset 0x028a:  [8-byte hash] [next data...]
```

**Order matches** the named-entry section — if named entries are `[NoBuildingStability, BuildingMaterialCostScale, CraftingCostScale]`, the secondary copy floats are in the same order. Use named entries to determine which float is which.

**Verification:** I edited Middle Eearth's secondary copy floats from `0.5/0.5` → `0.1/0.1` directly via Python. User launched the game and:
- World settings UI showed **0.1** (no longer 0.5) ✅
- Crafting cost was reduced to 1 material per item (0.1× rounded up) ✅
- Building cost similarly reduced ✅

**This is the real fix.** The named entries can probably be ignored entirely (or kept synced for safety; needs testing).

**Gielinor's structure is different (no count, no floats):**
```
Offset 0x01fc:  4c 5f 57 6f 72 6c 64 00     "L_World\0"
Offset 0x0204:  00                          1 byte padding
Offset 0x0205:  00 00 00 00                 count = 0 (NO entries!)
Offset 0x0209:  01 00 00 00                 version = 1
Offset 0x020d:  00 00 00 00                 4 bytes padding/zero
Offset 0x0211:  [8-byte hash] [next data...]
```

Gielinor was created BEFORE the custom difficulty feature shipped (per user). Its `CustomDifficultySettings` slot exists but is empty. To inject difficulty values into Gielinor would require **inserting 12 bytes of float data** (3 floats), which would shift every byte after offset 0x020d by 12. This is risky because the file is 3.3MB and may have offset tables/section size headers that depend on byte positions. **Phase 2 (Gielinor injection) was deprioritized in favor of Discovery #2 below.**

### Discovery #2 — `char_type` Field Unlocks Custom Worlds

**Problem:** User's main character `Serious_Beans` could only join standard worlds (like Gielinor). Couldn't join custom worlds (like Middle Eearth). When user creates a custom world in-game, the game forces creation of a NEW character that's tied to custom worlds.

**Root cause:** A single integer field in the character JSON: `meta_data.char_type`.

```
Serious_Beans (main, standard worlds only):  meta_data.char_type = 0
madood        (made when testing custom):    meta_data.char_type = 3
```

**The fix:** Edit `Serious_Beans.json` and change `char_type` from `0` to `3`. That's literally all.

## Specific Notes for Next Claude

### Key technical references
- **Secondary copy structure:** see "Discovery #1" above. `L_World\0` + 1 byte padding + uint32 count + uint32 version + N×float32 values. Order matches named entries.
- **char_type field:** `meta_data.char_type` in character JSON. 0 = standard-only confirmed. 3 = custom-compatible confirmed (whether also retains standard is OPEN).
- **Backups location:** `research_backups/` in project root (gitignored). Has all the safety copies created this session.
- **Game's autobackup behavior:** When the game saves, it COPIES the current .sav to .sav.backup THEN writes a fresh .sav. This is why our edits survived in the .backup file even when the game wrote new values to the main file. Useful for time-travel debugging.

### Hard-won lessons
- **The named-entry section of CustomDifficultySettings is a red herring.** It's serialized but the game doesn't read from it for runtime values. Always target the L_World+10 secondary copy.
- **Game completely re-serializes the file on every save** — file size changed from 197KB to 412KB to 413KB across sessions. The game doesn't preserve our binary edits in regions it cares about — it overwrites with its in-memory state. This means Phase 1 was doomed to fail at the named entries.
- **`research_backups/` is a discipline that paid off** — having pristine `.sav` copies plus `Middle Eearth - Copy.sav` from the user let us do byte-level differential analysis to find the secondary copy. ALWAYS make these.
- **Compact vs tab-indented JSON:** the game accepts both, but stick to tab-indented (`json.dump(..., indent="\t")`) for consistency with `parser.py` and the game's own format.

### Investigation toolkit (proven this session)
For any binary mystery, the workflow that worked:
1. Make pristine backups (×2)
2. Hex-dump regions of interest with both ASCII and offset
3. Search for known floats (e.g., `struct.pack('<f', 0.5)`) in the file to find candidate locations
4. Look for adjacent occurrences (suggests array structure)
5. Compare across versions (pristine vs edited vs game-saved) to spot which bytes the game touches
6. Cross-reference with the schema/manifest section if one exists (UE4 saves often have a field-name list separate from values)
