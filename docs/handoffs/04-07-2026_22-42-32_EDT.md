## Technical Notes (Key Carry-forward)

1. **SPUD is the save system.** Read the source at https://github.com/sinbad/SPUD before making any binary assumptions. The full chunk format is in `Source/SPUD/Public/SpudData.h`. Docs in `doc/tech.md`, `doc/props.md`, `doc/faq.md`, `doc/levelstreaming.md`.

2. **The full SPUD chunk tag list is in parser.py comments for reference.** Previously we were only searching for a hardcoded subset; now we know the complete set.

3. **Pces is a Dragonwilds-custom chunk.** Its contents follow Dragonwilds's own conventions, not SPUD's. But it lives *inside* SPUD-managed chunks (inside LEVL, inside LVLS).

4. **L_World is the master persistent world level.** The other `RSACell12800_X<N>_Y<N>` chunks are streaming cells for world chunks the player has visited. Player-built structures go in L_World, not the streaming cells.

5. **Per-piece data is fragmented across chunks.** A single cabin wall generates ~4 records: a Pces entry (placement metadata), plus NOBJ (persistent identifier) + PROP (property data) + CORA (transform/core state) for each of its sub-components. Transplanting one piece means copying all 4+ records, not just the Pces entry.

6. **SpudCache/L_World.lvl is a runtime working file.** Always clear it before loading a world whose .sav has been edited externally — otherwise SPUD may favor cached data over the edited save.

7. **World identity checks in Dragonwilds:**
   - Filename (e.g., `CabinMid.sav`)
   - Internal `WorldName` field in CINF chunk (must match filename)
   - Internal `WorldSlotName` field (same value as WorldName in our experiments)
   - **`WorldSaveGuid` (4 consecutive uint32 fields in CINF: GUID_A, GUID_B, GUID_C, GUID_D)** — must be unique across all saves or the game dedupes and hides duplicates

8. **WorldSaveGuid location in the file:** the GUID_A field starts at a specific offset in the CINF data block. In our test file CabinMid.sav, the 16-byte GUID was at offset `0x2ee` and appeared **2 times** in the file (we replaced both atomically via byte search-and-replace). Other saves may differ — use `find` on known GUID bytes to locate all occurrences, never hardcode the offset.
