### 6. BUGS.md tracker + ownership bug warning (commit `b353ec8`)
**Bug #001 reported by Accurious (RSDW Discord):** Converting a standard world to custom breaks ownership-bound features — locked chests, protection totems, privacy settings become unusable.

**Mitigation shipped:**
- Red inline warning on the Convert button in the World Data tab
- Detailed confirmation dialog on the Convert button listing exactly what to unlock/remove in-game BEFORE converting
- **`BUGS.md`** at project root with full bug #001 writeup: symptoms, workaround, suspected root cause (PROP offset table mismatch in CustomDifficultySettings field), fix plan for Phase 2 investigation

**Suspected root cause (from BUGS.md):** The PROP offset table has entries for fields 0..12. Field 8 = CustomDifficultySettings, fields 9-12 = LastSavedByEntries, PlayerOwnerGuid, GuidData, OwnerName. When we flip the mode byte inside CustomDifficultySettings, the loader may walk past its expected size and parse PlayerOwnerGuid bytes as difficulty data, corrupting the owner identification. Needs verification by reading PlayerOwnerGuid before/after conversion and comparing.

## 🎯 Next Session: Phase 2 — `.pak` Extraction

### Goal
Build the **GUID → item/quest name mapping** so the editor's Inventory, Quests, Spells, Mounts tabs can display real names instead of base64 strings.

### Expected outcome
When complete, the user's actual in-game items will show as e.g. "Wooden Training Sword" + its wiki icon, not as `AmuXOUitB08PYDCmdVA_Ug`. This is the multiplier effect — currently the entire edit UI works on unreadable IDs.

### Steps

**1. Locate the .pak file**
- User said game is at `E:\SteamLibrary\steamapps\common\RSDragonwilds`
- Find `*.pak` files in `<game>\Content\Paks\` or similar
- Note file size and encryption status

**2. Pick a tool**
Options:
- **FModel** (https://github.com/4sval/FModel) — Windows GUI, best UX, no encryption key issues for most UE5 games
- **UnrealPak.exe** — the official tool, ships with UE5 SDK
- **u4pak** (Python, https://github.com/panzi/u4pak) — CLI, good for automation
- **repak** (Rust, https://github.com/trumank/repak) — fast, CLI

**Recommendation:** Start with FModel for exploration (GUI makes it easy to find DataTables), then script the actual extraction with `repak` or `u4pak` if we want repeatable builds.

**3. Find the relevant DataTables**
Expected in `<game>\Content\Data\` or similar. Naming conventions:
- `DT_Item*.uasset` — item definitions (GUID → display name, mesh, icon reference)
- `DT_Quest*.uasset` — quest definitions
- `DT_Spell*.uasset` — spell definitions
- `DT_Mount*.uasset` — mount definitions
- `DT_Recipe*.uasset` — crafting recipes

**4. Parse `.uasset` files**
Options:
- FModel has built-in JSON export — easiest path
- Python `unreal_pak_parser` or `ue4parse` libraries
- Manual hex parsing (too much work)

**Recommendation:** Use FModel to export the DataTables to JSON, then write a Python script to convert those JSONs into our catalog format.

**5. Build the GUID → name mapping**
Output: `data/guid_map.json` with entries like:
```json
{
  "AmuXOUitB08PYDCmdVA_Ug": {
    "type": "quest",
    "name": "Ratcatcher",
    "catalog_id": "MQ03"
  },
  "xyz123...": {
    "type": "item",
    "name": "Wooden Training Sword",
    "catalog_id": "L001",
    "icon": "/static/images/items/L001.png"
  }
}
```

**6. Wire into the parser**
Extend `CharacterSave.get_summary()` to look up each inventory item's GUID in `guid_map` and include the friendly name + icon path in the summary. The UI then renders them automatically.

### Risks / blockers

- **Encryption:** If the .pak is AES-encrypted, we need the game's encryption key. Usually findable via FModel's AES helpers or in the game's executable.
- **Obfuscation:** Jagex may have obfuscated DataTable names, making them hard to find.
- **Game version drift:** When the game updates, the GUIDs may change or new items may appear. Need to document how to re-extract.
- **Disk space:** The .pak could be multiple GB. Only extract what's needed.

### What I should NOT do

- ❌ Try to read .pak files from my WSL environment without user involvement (I don't have the game files here)
- ❌ Start by writing my own parser — use an existing tool
- ❌ Extract the entire .pak to the repo — only the DataTables we need
- ❌ Commit the raw extracted .uasset files — only the derived JSON mappings

### What the user should prepare

- Have FModel installed (free, Windows, works on Steam game paths)
- Know where the game's Paks folder is
- Be ready to potentially try an AES key lookup if encrypted
