## 🎯 Major Discoveries This Session

### Discovery #1 — World Difficulty Enum at L_World+9

**Location:** uint32 LE at offset `L_World+9` in any world `.sav` file (find `b'L_World\x00'` byte sequence, then add 9).

**Values:**
| Value | Meaning | Effect |
|-------|---------|--------|
| `0` | Standard | Default — standard world classification |
| `1` | Hard | Hard difficulty preset (verified — user saw "Hard" in settings menu) |
| `2` | Unknown | Possibly Easy/Brutal/Nightmare — untested |
| `3` | **Custom** | World becomes "Custom World Settings" — exposes all 5 difficulty categories in UI, requires `char_type=3` characters to join |

**Verification:**
- Set Gielinor's byte from 0 → 1: in-game settings showed "Hard" difficulty (MODE still Standard)
- Set Gielinor's byte from 0 → 3: in-game settings showed "Custom World Settings" with Player/Crafting/Building/Environment/AI categories all unlocked
- madood (char_type=3) could see Gielinor in world list
- Serious_Beans (char_type=0) could NOT see Gielinor in world list — confirmed conversion worked at the access-control level
- User edited some custom settings via in-game UI successfully

### Discovery #2 — Game Reverts the Enum Byte on Save (PERSISTENCE PROBLEM)

**Sequence observed:**
1. We set Gielinor enum to 3 (file size: 3,375,805 bytes)
2. User launched game, picked madood, joined Gielinor
3. Game loaded Gielinor as Custom (UI confirmed)
4. User edited custom difficulty settings
5. Game auto-saved on exit (or during play)
6. **Save process overwrote our byte:** moved current `.sav` (with `enum=3`) to `.sav.backup`, wrote fresh `.sav` with `enum=0`
7. New live file is **+11,064 bytes larger** (3,386,869 bytes)
8. .sav.backup STILL has `enum=3` (preserved as expected)

**What the game added (11KB growth):**
- 2 new named TagName/NameProperty entries: `BuildingMaterialCostScale` and `CraftingCostScale` (Gielinor previously only had `FriendlyFire`)
- 12 bytes of zeros at the secondary copy float region (allocated space for 3 floats — same structure as Middle Eearth)
- Schema reordering in the field-name list
- Other state changes

**Implication:** The game **permanently grew Gielinor's structure to be custom-capable**, but its in-memory state when serializing wrote `enum=0`. The byte we found is either:
- (A) A UI cache / display value, with the real persistent storage elsewhere
- (B) The real storage, but the game has logic that resets it on save based on character/session state
- (C) The real storage, and the game's serializer just always writes 0 for this enum unless certain conditions are met

**Critical observation:** Gielinor is now structurally identical to Middle Eearth (3 named entries + 12-byte float space). If we re-flip the enum to 3 NOW, it might persist this time because the structure is consistent. **This is the open question for next session.**

### Discovery #3 — `char_type` Mutual Exclusivity Confirmed

## 🚨 Hard-Won Lessons This Session

1. **The game's save process is destructive to manual edits.** Any byte you flip can be overwritten the next time the game writes the file. ALWAYS verify edits are still in place after the user plays.
2. **The game can grow file structures automatically.** Loading Gielinor in custom mode caused the game to add BMS+CCS schema entries permanently. This is a one-way change — those entries persist even after enum reverts.
3. **`.sav.backup` is the previous version, not "earlier today's pristine state."** Each game save updates it. Use research_backups/ for true historical state.
4. **L_World position shifts when file grows.** Always re-find via byte search; never hardcode the offset.
5. **Difficulty enum and World Mode are different concepts in the UI.** Standard/Hard/Custom is the difficulty PRESET. Standard/Custom World Settings is the MODE classification. They appear linked through the enum byte but might have separate persistence.
6. **The game's UI shows different categories based on world type.** Standard worlds expose only Environment > FriendlyFire. Custom worlds expose Player, Crafting, Building, Environment (full), and AI.
