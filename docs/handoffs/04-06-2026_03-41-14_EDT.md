## Session Summary

This was the **first session** of the project. We built a Flask web-based save editor for RuneScape: Dragonwilds from scratch — going from "what is this binary file" through full reverse engineering of two save formats (character JSON + world binary), to a functional editor covering 95% of practical save editing needs. ## What works (confirmed end-to-end via Flask test client)

### Character editing (`SaveCharacters/*.json` — plain JSON, completely safe)
- **Vitals:** Health, Stamina, Sustenance, Hydration, Toxicity, Endurance (uses correct field names like `SustenanceValue`, NOT `CurrentValue`)
- **Skills:** All 10 skills' XP with live level/progress bar UI, MAX button per skill
- **Inventory:** Edit `Durability` for gear, `Count` for stackables, delete items, "Repair All" and "Max All Stacks" mass actions
- **Loadout:** Edit equipment slot durability; properly handles slots 5-8 which reference inventory by index instead of storing items directly
- **Quests:** Toggle quest state (Not Started / In Progress / Completed), per-quest expandable editor for individual `QuestBools` and `QuestInts`, "Complete All Quests" button
- **Status Effects:** Clear individual effects (Cold, Poison, Burning, Bleeding, Slow, Wither) or "Clear All Debuffs"
- **Position / Teleport:** Edit X/Y/Z coords (parses and reformats the `V(X=..., Y=..., Z=...)` string)
- **Spell Loadout:** All 48 slots editable, dropdowns populated from `SpellsUnlocked`, "Fill All 48" with selected spell, "Clear All"
- **Mounts:** List unlocked, equip, remove, add by ID (no catalog yet — manual ID entry)
- **Map / Fog of War:** "Reveal Full Map" sets `RevealedRegionsBitmap` to 0xFFFFFFFF
- **Hardcore mode toggle**
- **"Full Restore" mega-button** — heals all vitals + clears all debuffs in one click

### World editing (`SaveGames/*.sav` — binary, partially editable)
- **89 World Storage Containers** — edit `Count` of stackable items, `Durability` of gear inside chests (length-preserving JSON rewrites with whitespace padding)
- **Weather** — change type (Sunny/Cloudy/Rain/Storm/Snow/Fog/Hail/Wind) and remaining time per region (2 regions: `base`, `dowdun`)
- **24 World Events** — toggle individual event triggers (raids, ambushes), "Disable All Raids/Ambushes" button (sets cooldowns to 999 days)
- **Custom Difficulty Settings (Phase 1)** — read existing entries from binary `WorldSaveSettings` block, edit float values in place (length-preserving 4-byte float swap). **Works at file level but does NOT apply in-game** — see "What didn't work" below.

### Infrastructure
- **Auto-backup** — every save creates a timestamped copy in `editor_backups/` next to the original
- **Multi-character support** — character cards in header, `?char=<filename>` URL param
- **Reload Files** button — re-discovers saves from disk (refreshes in-memory state)
- **Per-world Save World button** — only appears after edits to that world

## What we discovered (the reverse engineering)

### Save file locations
```
%LOCALAPPDATA%\RSDragonwilds\Saved\
├── SaveCharacters\<Name>.json          ← PLAIN JSON character data
├── SaveGames\<World>.sav                ← Binary world data with embedded JSON sections
└── Logs\*.log                           ← Useful for finding GameplayTag names
```

**Critical insight:** Character data and world data are stored in **separate files**. Editing characters does not require touching the binary world file at all.

### Character JSON structure (everything in plain JSON, ~64KB)
Top-level keys: `Version`, `meta_data`, `SaveCount`, `Character`, `Inventory`, `PersonalInventory`, `Loadout`, `Progress`, `QuestProgress`, `RevealedLandmarks`, `RevealedFog`, `Skills`, `Journal`, `MapCustomization`, `Spellcasting`, `Hardcore`, `Backup`

**Field name gotchas (we hit these and they cost us time):**
- Sustenance uses `SustenanceValue`, NOT `CurrentValue`
- Hydration uses `HydrationValue`
- Toxicity uses `ToxicityValue` (also `HighestToxicityValue`)
- Endurance uses `EnduranceValue`
- Health and Stamina DO use `CurrentValue` (inconsistent!)
- Inventory stack count is `Count`, NOT `Quantity`
- Position is a STRING in format `V(X=11924.46, Y=184569.45, Z=-3183.06)` not numeric coords
- The `Backup` field at the bottom of the JSON looks like a CRC — we don't touch it and the game seems to accept edited files anyway

**Skill ID mappings (reverse-engineered from a known character):** 10 skills mapped — see `parser.py` `SKILL_NAMES` dict.

### World binary format
- Jagex "Dominion Engine" — custom UE5 fork
- Section markers: `SAVE` `INFO` `CINF` `GLOB` `METAL` `VERS` `CNIX` `CLST` `CDVE` `CDEF` `PNIX` `GOBS` `NOBJ` `PROP` `GLAI`
- 5,893 named game objects (`NOBJ` entries) in a sample 3.3MB save: ore nodes, spawners, NPCs, trees, treasure chests, etc. — all in binary UE4 property format, NOT currently editable
- **168 embedded JSON sections** for component data (containers, stations, events, weather) — these ARE editable
- Editing strategy: length-preserving rewrites with whitespace padding. If new JSON > original, fall back to compact format. If still too big, skip with a warning.

### The custom difficulty system (the big discovery)
The game has 35 customizable difficulty tags exposed via the world creation UI:
- **32 combat tags** — 11 enemy types (Beast, Boss, Construct, Critter, Garou, Goblin, MiniBoss, Skeleton, Undead, Zamorak) × 3 stats each (Damage, Health, Resistances) + DisableAggressiveAI
- **3 progression/environment/player toggles** — FriendlyFire, NoBuildingStability, BuildingMaterialCostScale, CraftingCostScale

**How we found the full list:** Mined the game's `*.log` files in `Saved/Logs/` (the engine logs all GameplayTag names at startup).

**Binary layout:** Each entry in the `CustomDifficultySettings` array is stored as:
```
[FString "TagName"]                  ← length-prefixed (8 bytes)
[FString "NameProperty"]             ← length-prefixed (13 bytes)
[~13 bytes of UE4 property metadata]
[FString <difficulty tag>]           ← variable
[FString "None"]                     ← 9 bytes (terminator)
[float32 <value>]                    ← LITTLE ENDIAN, 4 bytes
```

Confirmed values from `Middle Eearth.sav` (a world the user created with custom settings):
- `Difficulty.Player.NoBuildingStability` = 1.0
- `Difficulty.Progression.BuildingMaterialCostScale` = 0.5
- `Difficulty.Progression.CraftingCostScale` = 0.5

`Gielinor.sav` (the user's main world) has only 1 active entry: `Difficulty.Environment.FriendlyFire` = 0.0. The other 34 tags aren't stored at all.

### What's NOT customizable (and why "double mining yields" is impossible via save edit)
There is **NO gather yield / mining / woodcutting / monster drop multiplier** in the game's custom difficulty system. The 35 tags above are everything. To actually multiply gather output you'd need to mod the game's `.pak` files (DataTable mods like `DT_OreNode_*`, `DT_LootTable_*`) — that's a separate, much larger project requiring UnrealPak/Fmodel + a UE5 mod loader. **Workarounds in the editor:** lower `CraftingCostScale` (need less material to craft), lower AI Health (faster kills = more drops/hour), or just edit inventory `Count` directly to 999.

## What didn't work (and what we don't yet understand)

### Phase 1 difficulty edit doesn't apply in-game
**Test scenario:**
1. User edited `Difficulty.Progression.CraftingCostScale` to ~0 in Middle Earth via the editor
2. Clicked "Save World" — editor wrote the file (size unchanged: 197,679 bytes)
3. User launched game and went into Middle Earth's settings/edit menu
4. **Game showed the original 0.5 value, not the edited value**
5. User played briefly, then quit
6. After quit: file size **doubled** (197KB → 412KB), values back to 0.5/0.5/1.0

**What this means:** The game either ignored our edit when reading, OR overwrote it when saving. We don't know which yet.

**Possible causes (in order of likelihood):**
1. Game caches difficulty values at world load and writes the cached values back on save (overwriting our edit)
2. Hidden checksum/CRC over the difficulty block — game detects tampering and falls back to a stored copy
3. Secondary copy of the values somewhere else in the file that takes precedence at runtime
4. Game restored from `.sav.backup` because our edit failed validation

**Next session must test this cleanly** — see Priority 1 in "Next Session" below. The test procedure is in `TODO_TOMORROW.md`.

## Specific notes for next Claude

### Hard-won lessons
- **Always close the game before editing world saves.** The game holds a lock and caches state in memory. The user just verified that even after clean save+launch, the game appears to overwrite our edits.
- **Sustenance/Hydration/Toxicity/Endurance use `<Stat>Value` field names**, not `CurrentValue`. Don't trust pattern matching across stats — we got bitten by this on the first version of the editor.
- **Inventory items use `Count` not `Quantity`.** Same bite.
- **Position is a string `V(X=..., Y=..., Z=...)`.** Parse with regex, format as `V(X={x:.6f}, Y={y:.6f}, Z={z:.6f})`. Watch out for `V(0)` which means "no position recorded yet".
- **The Flask server caches saves in memory.** Edits go to the in-memory `CharacterSave`/`WorldSave` instances. The "Reload Files" button re-discovers from disk. After the user plays the game and quits, the in-memory state is STALE — they need to click Reload to see the new disk values. (We saw this confuse the user this session — "the editor shows 0 but the game shows 0.5" was actually "stale memory shows 0, disk and game show 0.5".)

### Architectural decisions made this session
- **Two-class parser design** (`CharacterSave` vs `WorldSave`) because the formats are completely different. Don't try to unify them.
- **Length-preserving world saves** — when a JSON section grows, pad with whitespace inside the JSON. When it would shrink, also pad. When even the compact form is too big, skip with a warning. This avoids needing to update binary section size headers (which are unknown territory).
- **In-memory edit queue** in the JS — the user makes a change, it queues, queue is POSTed to `/update`, on success the dirty flag goes up. The Save button only commits to disk when the user explicitly clicks it. This lets the user batch many edits before writing.
- **Per-feature edit actions in the API** rather than full document replacement — keeps the surface area small and lets the parser validate each action.
- **Difficulty settings catalog hardcoded** in `parser.py` `KNOWN_DIFFICULTY_TAGS` based on logs. If the game adds new tags in an update, this list will need to be re-mined from logs.
