## Technical Notes (Key Carry-forward)

1. **SPUD binary format has empirical quirks not in the header files:**
   - LEVL body: 8-byte version header `0a 02 00 00 f9 03 00 00` (constants 522 and 1017) comes BETWEEN the Name FString and the first sub-chunk. SpudData.h doesn't document this.
   - NOBJ body: ClassID uint32 is at the START of the body, not the end. SpudData.h's `FSpudObjectData` struct lists ClassID last but the serialization order appears reversed.

2. **`0a 02 00 00 f9 03 00 00` is a universal SPUD marker.** Appears at the start of Pces records (at offset 0x44), inside LEVL headers, inside NOBJ headers. Anywhere SPUD stores versioned object data, this 8-byte sequence shows up. Safe to use as a structural anchor.

3. **GlobalBuildingManager is the sole container for all player-built piece state.** Not per-piece NOBJs. Not SPWN records. Everything goes through this one actor's CUST data. Makes surgical transplant much simpler.

4. **Pces chunk records are preserved byte-for-byte across worlds** (same seed → same class GUIDs → no translation needed). The only identity tied to a specific world is the NOBJ's UAID suffix, which is world-local but doesn't affect cabin pieces (they reference each other via base64 GUIDs inside the Pces chunk, not via actor UAIDs).

5. **Chunk length chain for surgical transplant** — adding N bytes of Pces data means updating these headers (in order):
   - Pces chunk `length` field (+N)
   - CUST TArray count uint32 (+N)
   - CUST chunk `length` field (+N, if the TArray count counts toward chunk length — it does since TArray serializes as count+bytes within the chunk body)
   - NOBJ chunk `length` field (+N)
   - LATS chunk `length` field (+N)
   - LEVL chunk `length` field (+N)
   - LVLS chunk `length` field (+N)
   - SAVE chunk `length` field (+N)

   That's 8 length fields. Each is a 4-byte uint32 LE at a known offset relative to its chunk's 4-byte tag. Implementation is mechanical once we have all 8 offsets.

6. **Next-session gotcha to watch for:** GlobalBuildingManager's PROP chunk has 4 counter properties (53 B of PROP data). These might include a "total pieces" counter that needs incrementing when we add cabin pieces. If our surgical transplant works structurally but the cabin still doesn't render, the counters in PROP are the prime suspect. Test: if surgical fails with "Persistence ID ''" error again, diff D's and E's GBM PROP bytes to see if any counter values changed.
